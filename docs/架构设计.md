## 一、项目定位

面向 **Polymarket 短周期二元预测市场**的自动化交易机器人，技术栈 **Rust + Tokio + Cargo Workspace**。

### 策略本质

> 在 Up / Down 两边反复挂 Maker 低买单，让 `min(Q_up, Q_down) > C_total`，无论哪边交割胜出都净赚。

这是一个「做空波动率 / 吃震荡」策略：
- 震荡盘：两边都接到低位货 → 盈利。
- 单边趋势盘：只接到注定输的一边 → 瘸腿敞口 → 对冲减损。

## 二、整体架构

### 数据流

```
                  ┌─────────────────────────────────────────────┐
                  │              事件循环 (单写者)                  │
   行情/成交事件    │                                              │   下单/撤单指令
  ┌──────────┐    │   Event ──► [账本更新] ──► [策略决策] ──► Command  │   ┌──────────┐
  │ 执行后端  │───►│                            │                 │──►│ 执行后端  │
  │ (trait)  │    │                            ▼                 │   │ (trait)  │
  └──────────┘    │                      [风控审计器]             │   └──────────┘
                  │                     (Cash Guard)              │
                  └─────────────────────────────────────────────┘
                                          │
                                          ▼ 每笔变更原子落盘
                                    ┌──────────┐
                                    │  WAL 日志  │
                                    └──────────┘
```

### 关键架构决策

1. **单写者串行**：所有状态变更经过唯一的 Engine 事件循环，无锁无竞态。
2. **策略产出指令列表**：策略不直接调用交易所，只产出 `Vec<Command>`，由 engine 下发。
3. **执行后端 trait**：策略与执行解耦，可切换模拟 / 真实后端。
4. **世代号竞态隔离**：每批挂单携带递增世代号，成交一律入账，旧世代成交不触发重算。
5. **数值用 `rust_decimal`**：金额/价格/股数统一十进制精确运算，禁止 f64。

## 三、Crate 划分（9 个，单向依赖）

```
app → backtest/engine → {strategy→fsm, risk, ledger, exchange} → domain
```

| Crate | 职责 |
|-------|------|
| **domain** | 核心领域模型（数值类型、Side、Order/Fill/Command、世代号、PnL/EV、FeeModel、MarketSnapshot、RobotState）。纯逻辑零 IO |
| **ledger** | 账本（双边持仓/成本/均价）+ WAL 原子落盘与重启复原 |
| **fsm** | 有限状态机：6 状态流转 + 对冲触发判定，纯逻辑 |
| **strategy** | 策略决策：梯度接低（布阵+配对重算）+ 对冲决策（HedgePhase/HedgeDecision/decide_hedge_step）+ 对冲执行（hedge_taker_step） |
| **risk** | 风控审计：Cash Guard（cash_guard_ratio ≥ reserve 校验）+ 三资金池 |
| **exchange** | 执行后端 trait + Simulator 模拟撮合（Maker 保守填单 + Taker 即时成交） |
| **engine** | 单写者事件循环：事件分发 + 记账 + 限流（hedge_fired_this_tick）+ 预算追踪（hedge_budget_remaining）+ 风控 |
| **backtest** | 验证工具：加载真实 CSV 行情 + 跑回测产出 JSON 报告 |
| **app** | 可执行入口 + 配置加载 + 日志 |

### 模块职责边界

| 想看什么 | 去哪看 |
|----------|--------|
| 策略怎么决策（布阵/配对/对冲选侧） | strategy + fsm |
| 事件怎么串起来 | engine |
| 资金怎么分配和保护 | risk |
| 成交怎么撮合 | exchange |

## 四、回测与验证体系

### 数据源

- `logs/` 目录：933 个真实逐秒盘口 CSV（BTC 15 分钟市场，2026-05~06）
- 胜负判定：末个有效 `pm_price > ptb` → Up 赢

### 回测流程

```bash
# Step 1: Rust 跑回测,记录逐笔成交,产出 JSON
cargo run -p backtest --example report_json --release

# Step 2: Python 渲染 HTML 逐场报告
python3 analysis_data/render_rust_report.py
```

产出：`analysis_data/output/rust_strategy/full_analysis.json` + 逐场 HTML 报告。

### analysis_data 工具

| 文件 | 作用 |
|------|------|
| `fetch.py` | 从 Polymarket Data API 拉取真实玩家交易记录 |
| `settlement.py` | 从 CLOB API 获取结算数据 |
| `analyze.py` | 按 condition_id 分组重建场次 |
| `run_analysis.py` | 逐场 PnL 模拟 + HTML 报告渲染（含角色列 Maker/Taker、方向列颜色） |
| `render_rust_report.py` | 从 Rust 回测 JSON 渲染 HTML（复用 run_analysis 渲染函数） |

## 五、资金池分层结构

总资金 $V$ 的完整分层关系如下图。一级切分由 `CapitalPools` 管理，二级切分由各模块配置参数控制。

```
总资金 V (100%)
├── 备用金池 Reserve (25%)
│     └── 日常禁动；Cash Guard 红线 ≥ 此比例（默认等于 reserve）
│
├── 核心做市池 Grid_Maker (52.5%)
│     ├── 初始布阵三层梯度（strategy LadderRung.pool_fraction）
│     │     ├── 1 层：pool_fraction = 2%  → 实际占总资金 52.5% × 2% = 1.05%
│     │     ├── 2 层：pool_fraction = 3%  → 1.575%
│     │     └── 3 层：pool_fraction = 5%  → 2.625%
│     └── 续挂追低（strategy follow_fraction = 2%）
│           └── 每次成交后追低一档，占做市池 2%
│
└── 动量对冲池 Hedge_Attack (22.5%)
      └── 对冲 Taker 单步（engine hedge_step_fraction = 20%）
            └── 每步预算 = 对冲池 × 20% = 总资金的 4.5%
            └── 对冲池剩余额度跨阶段不重置（hedge_budget_remaining）
```

### 关键约束

| 约束 | 说明 |
|------|------|
| 三池比例之和 = 1 | `CapitalPools::new` 构造时 assert |
| cash_guard_ratio ≥ reserve 比例 | `RiskAuditor::new` 构造时 assert |
| pool_fraction / follow_fraction 是相对 Grid_Maker 池 | 不是相对总资金 V |
| hedge_step_fraction 是相对 Hedge_Attack 池 | 不是相对总资金 V |
| 对冲不触碰备用金池 | 对冲预算来源仅 Hedge_Attack 池额度 |
| 对冲池跨阶段不重置 | Dynamic→Ev 步数归零但剩余额度不归零 |

### 参数索引

| 参数 | 位置 | 默认值 | 相对基准 |
|------|------|--------|----------|
| `PoolRatios.reserve` | `risk::pool` | 0.25 | 总资金 V |
| `PoolRatios.grid_maker` | `risk::pool` | 0.525 | 总资金 V |
| `PoolRatios.hedge_attack` | `risk::pool` | 0.225 | 总资金 V |
| `cash_guard_ratio` | `risk::auditor` | = reserve | 总资金 V |
| `LadderRung.pool_fraction` | `strategy` | 0.02 / 0.03 / 0.05 | Grid_Maker 池 |
| `LadderConfig.follow_fraction` | `strategy` | 0.02 | Grid_Maker 池 |
| `EngineConfig.hedge_step_fraction` | `engine` | 0.20 | Hedge_Attack 池 |
| `EngineConfig.max_taker_steps` | `engine` | 5 | 每对冲阶段独立计数 |

## 六、关键模块说明

### 6.1 domain

| 模块 | 内容 |
|------|------|
| `types.rs` | `Price`/`Qty`/`Money`(Decimal 别名)、`Side`(Up/Down)、`OrderRole`(Maker/Taker) |
| `fee.rs` | `FeeModel`：费率可配，`net_qty()` 算净到手股数 |
| `pnl.rs` | `PositionSnapshot`：条件 PnL + EV + 利润锁定 + `breached_side()` + `qty(side)` |
| `order.rs` | `Generation`/`OrderId`/`Order`/`Fill`/`Command`/`OrderConstraints` |
| `market.rs` | `BookTop`/`MarketSnapshot`/`mark_price()`(中间价,缺失回退) |
| `state.rs` | `RobotState` 6 状态枚举 |

### 6.2 strategy 策略决策

- **做市**：`GradientLadder`（deploy/select_main_field/recompute_after_fill/hedge_taker_step）
- **对冲决策**：`HedgePhase`（None/Dynamic/Ev）+ `HedgeDecision`（Skip/Execute）+ `decide_hedge_step()`
- `advantaged_side()`：按 mark_price > 0.5 找优势方

### 6.3 engine 编排

- 事件分发（BookUpdate/Filled/Rejected/Canceled）
- `sync_hedge_phase()`：阶段切换检测 + 步数重置
- `hedge_step()`：限流检查 → 调 strategy 要决策 → 执行
- `hedge_fired_this_tick`：每 tick 最多 1 步 Taker
- `hedge_budget_remaining`：对冲池剩余，跨阶段不重置
- `approve_command()`：Cash Guard 风控

### 6.4 exchange Simulator

- Maker：进入挂单簿，卖一价严格穿越限价才成交（保守口径）。
- Taker：submit 瞬间以 best_ask 即时成交，吃不到则拒单。
- `build_fill(order, exec_price)`：统一构造成交回报（含 role），费率按角色扣股数。
