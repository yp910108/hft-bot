## 一、项目定位与核心结论

本项目是面向 **Polymarket BTC 15 分钟周期二元预测市场**的自动化高频交易机器人，技术栈为 **Rust + Tokio**。

### 1.1 策略本质（必须先理解）

剥开所有术语，策略的核心是一句话：

> **在 Up / Down 两边反复挂 Maker 低买单，试图让 `min(Q_up, Q_down) > C_total`，从而无论哪边交割胜出都净赚。**

因为二元市场里 `ask_up + ask_down` 恒 ≥ 1，**两边的便宜筹码必然来自不同时刻**，所以凑出双向盈利**必须依赖价格在 15 分钟内来回震荡**。

**因此结论是：这是一个「做空波动率 / 吃震荡」策略。**
- 震荡盘：两边都能接到低位货 → 盈利。
- 单边趋势盘：只接得到注定输的那边 → 形成瘸腿敞口 → 触发各级对冲（本质是减损机制，不是赚钱引擎）。

**真正的利润来源只有震荡盘的 Maker 价差；整套对冲是趋势盘里的止损机制。** 策略长期期望能否为正，取决于「震荡盘赚的」能否覆盖「趋势盘亏的 + 摩擦成本」——这一点策略文档未论证，是**最大的开放问题**，必须靠回测/纸面交易用真实数据验证后才能下结论。

### 1.2 三大推进决策（已与用户确认）

1. **回测/纸面交易优先**：edge 未证明，先用历史/合成数据验证期望为正，再考虑实盘。架构必须把「策略逻辑」与「执行后端」彻底解耦。
2. **策略修复先逐条讨论确认再写代码**：见第四节 9 项风险清单。
3. **先不接真实交易所，先做模拟**：行情/连接层先做 trait 抽象 + 模拟实现，真实 Polymarket CLOB 接入后置。

## 二、整体架构

### 2.1 数据流

```
                  ┌─────────────────────────────────────────────┐
                  │              事件循环 (单写者)                  │
   行情/成交事件    │                                              │   下单/撤单指令
  ┌──────────┐    │   Event ──► [账本更新] ──► [策略状态机] ──► Command   │   ┌──────────┐
  │ 执行后端  │───►│                            │                 │──►│ 执行后端  │
  │ (trait)  │    │                            ▼                 │   │ (trait)  │
  └──────────┘    │                      [风控审计器]             │   └──────────┘
       ▲          │                     (Cash Guard/红线)         │        │
       │          └─────────────────────────────────────────────┘        │
       │                                  │                               │
       │                                  ▼ 每笔变更原子落盘                 │
       │                            ┌──────────┐                          │
       └────────────────────────────│  持久化   │◄─────────────────────────┘
          重启时从冷数据复原           │  (WAL)   │
                                    └──────────┘
```

核心思路：**执行后端既是事件来源（行情/成交），也是指令去处（下单/撤单）**。策略本身完全不知道自己跑在回测还是实盘——它只面对 trait。

### 2.2 两个贯穿全局的设计原则

1. **执行后端可替换**（落地决策①③）：用 trait 把「策略逻辑」和「跑在哪」解耦，同一套策略代码零改动跑在 `回测 / 模拟 / 真实 Polymarket` 三种后端上。
2. **单写者事件循环**（修复 #6 竞态）：所有状态变更串行经过一个事件循环，杜绝并发改账本。

## 三、Crate 划分与依赖

纯 workspace（根目录无 `[package]`），8 个 crate，依赖**严格单向**：

```
app → engine → {strategy → fsm, risk, ledger, exchange} → domain
                                                            ↑
                          所有 crate 都只依赖 domain，domain 不依赖任何业务 crate
```

`domain` 是地基且零 IO 依赖，所以 PnL/EV/费用这些最关键的数学逻辑能用纯单元测试覆盖。

| Crate | 职责 | 对应策略章节 |
|-------|------|------------|
| **domain** | 核心领域模型（数值类型、Side、Order、Fill、世代号、PnL/EV、FeeModel、Mark Price、状态枚举）。纯逻辑零 IO | 全局词汇表 |
| **ledger** | 账本：持仓/成本/均价，原子落盘 + 重启复原 | §八 高可用账本 |
| **fsm** | 有限状态机：6 状态流转 + 对冲触发判定，纯逻辑 | §三 FSM 架构 |
| **strategy** | 策略执行：梯度接低 / 动态对冲 / EV 对冲，把 FSM 决策翻译成下单指令 | §四~六 |
| **risk** | 风控审计：Cash Guard + 三资金池 + 红线约束 | §一、§八 |
| **exchange** | 执行后端 trait + 回测 / 模拟 / 真实三实现 | 行情/连接层 |
| **engine** | 单写者事件循环，串联以上模块 | §三 事件驱动 |
| **app** | 可执行入口（bin `hft-bot`）+ 配置加载 + 日志 | — |

### 3.1 关键地基决策：数值类型

金额、价格、股数统一使用 `rust_decimal::Decimal`（10 进制定点数），**禁止用 f64 表示金钱量**。
- 原因：价格在 0~1 间且反复做「成本 ÷ 股数」除法，f64 会累积误差；Decimal 十进制精确运算可避免，无需靠舍入压制浮点噪声。
- domain 层定义语义别名：`Price` / `Qty` / `Money` 均为 `Decimal`。

### 4.1 策略遗漏的核心业务逻辑（开发中补齐）

1. 行情接入与订单簿维护（REST 下单 + WebSocket 行情、Mark Price 计算）
2. 订单生命周期：下单/撤单确认、超时、拒单、部分成交
3. 「新一场开始」识别：发现新一轮、拿到新市场 token id、上轮持仓交割入账
4. 网络/API 异常：断线重连、下单失败重试、与交易所对账
5. 持久化具体形态：WAL / 事件日志、复原流程、与交易所对账关系
6. 回测/纸面交易框架（最优先，验证 edge）

### 6.2 ledger（内存账本，10 单测）

- `Ledger`：聚合 Up/Down 两侧 `SideBook`。
- `apply_fill()`：买入累加净入仓股数与花费现金（手续费已在撮合层体现为股数扣减）；卖出按加权平均成本法扣减、持仓清零时重置，不沉淀已实现利润（符合策略「唯一账本」原则）。
- `average_price(side)`：加权均价，返回**精确值**不做舍入（均价为派生统计量，决策一律基于原始成本与股数；4 位卡死要求已废弃，详见策略.md §八批注）。
- `total_cost()` / `snapshot()`：产出 `PositionSnapshot` 直接喂给 domain 的 PnL 计算。
- 已覆盖：空账本、单笔买入、Taker 扣股数（手续费体现为净股数减少而非额外成本）、多笔加权均价、均价保留精确值不舍入、双边独立、卖出保持均价、清仓重置、超卖钳制、快照联动 PnL。
