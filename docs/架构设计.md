## 一、项目定位

面向 **Polymarket 短周期二元预测市场**的自动化交易机器人，技术栈 **Rust + Tokio + Cargo Workspace**。

### 策略本质

> 在 Up / Down 两边反复挂 Maker 低买单，让 `min(Q_up, Q_down) > C_total`，无论哪边交割胜出都净赚。

这是一个「做空波动率 / 吃震荡」策略：
- 震荡盘：两边都接到低位货 → 盈利。
- 单边趋势盘：只接到注定输的一边 → 瘸腿敞口 → 对冲减损。

## 二、整体架构

### 数据流

```
                  ┌─────────────────────────────────────────────┐
                  │              事件循环 (单写者)                  │
   行情/成交事件    │                                              │   下单/撤单指令
  ┌──────────┐    │   Event ──► [账本更新] ──► [策略状态机] ──► Command   │   ┌──────────┐
  │ 执行后端  │───►│                            │                 │──►│ 执行后端  │
  │ (trait)  │    │                            ▼                 │   │ (trait)  │
  └──────────┘    │                      [风控审计器]             │   └──────────┘
                  │                     (Cash Guard)              │
                  └─────────────────────────────────────────────┘
                                          │
                                          ▼ 每笔变更原子落盘
                                    ┌──────────┐
                                    │  WAL 日志  │
                                    └──────────┘
```

### 关键架构决策

1. **单写者串行**：所有状态变更经过唯一的 Engine 事件循环，无锁无竞态。
2. **策略产出指令列表**：策略不直接调用交易所，只产出 `Vec<Command>`，由 engine 下发。
3. **执行后端 trait**：策略与执行解耦，可切换模拟 / 真实后端。
4. **世代号竞态隔离**：每批挂单携带递增世代号，成交一律入账，旧世代成交不触发重算。
5. **数值用 `rust_decimal`**：金额/价格/股数统一十进制精确运算，禁止 f64。

## 三、Crate 划分（9 个，单向依赖）

```
app → backtest/engine → {strategy→fsm, risk, ledger, exchange} → domain
```

| Crate | 职责 |
|-------|------|
| **domain** | 核心领域模型（数值类型、Side、Order/Fill/Command、世代号、PnL/EV、FeeModel、MarketSnapshot、RobotState）。纯逻辑零 IO |
| **ledger** | 账本（双边持仓/成本/均价）+ WAL 原子落盘与重启复原 |
| **fsm** | 有限状态机：6 状态流转 + 对冲触发判定，纯逻辑 |
| **strategy** | 策略执行：梯度接低（布阵+跨侧配对重算）+ 对冲 Taker 步进 |
| **risk** | 风控审计：Cash Guard + 三资金池 |
| **exchange** | 执行后端 trait + Simulator 模拟撮合（Maker 保守填单 + Taker 即时成交） |
| **engine** | 单写者事件循环，串联以上模块；含对冲状态分派与 Taker 计步 |
| **backtest** | 验证工具：加载真实 CSV 行情 + 跑回测产出 JSON 报告 |
| **app** | 可执行入口 + 配置加载 + 日志 |

## 四、回测与验证体系

### 数据源

- `logs/` 目录：933 个真实逐秒盘口 CSV（BTC 15 分钟市场，2026-05~06）
- 胜负判定：末个有效 `pm_price > ptb` → Up 赢

### 回测流程

```bash
# Step 1: Rust 跑回测,记录逐笔成交,产出 JSON
cargo run -p backtest --example report_json --release

# Step 2: Python 渲染 HTML 逐场报告
python3 analysis_data/render_rust_report.py
```

产出：`analysis_data/output/rust_strategy/full_analysis.json` + 逐场 HTML 报告。

### analysis_data 工具

| 文件 | 作用 |
|------|------|
| `fetch.py` | 从 Polymarket Data API 拉取真实玩家交易记录 |
| `settlement.py` | 从 CLOB API 获取结算数据 |
| `analyze.py` | 按 condition_id 分组重建场次 |
| `run_analysis.py` | 逐场 PnL 模拟 + HTML 报告渲染 |
| `render_rust_report.py` | 从 Rust 回测 JSON 渲染 HTML（复用 run_analysis 渲染函数） |

## 五、关键模块说明

### 5.1 domain

| 模块 | 内容 |
|------|------|
| `types.rs` | `Price`/`Qty`/`Money`(Decimal 别名)、`Side`(Up/Down)、`OrderRole`(Maker/Taker) |
| `fee.rs` | `FeeModel`：费率可配，`net_qty()` 算净到手股数 |
| `pnl.rs` | `PositionSnapshot`：条件 PnL + EV + 利润锁定判定 |
| `order.rs` | `Generation`/`OrderId`/`Order`/`Fill`/`Command`/`OrderConstraints`(最小量+精度量化) |
| `market.rs` | `BookTop`/`MarketSnapshot`/`mark_price()`(中间价,缺失回退) |
| `state.rs` | `RobotState` 6 状态枚举 |

### 5.2 engine 对冲机制

- `HedgePhase` 边沿检测（粗粒度 None/Dynamic/Ev），阶段变化时重置 Taker 计数。
- 对冲推进纯成交驱动：Filled 事件触发 `hedge_step()`，BookUpdate 不发 Taker。
- `lame_side()` 取 PnL 穿透亏损线的一侧；`advantaged_side()` 按 mark price > 0.5。
- DynamicHedging 与 EvHedging 各自独立 `max_taker_steps=5`。

### 5.3 exchange Simulator

- Maker：进入挂单簿，卖一价严格穿越限价才成交（保守口径）。
- Taker：submit 瞬间以 best_ask 即时成交，吃不到则拒单。
- `build_fill(order, exec_price)`：统一构造成交回报，费率按角色扣股数。
