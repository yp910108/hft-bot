## 回测验证结论

933 个真实历史市场回测（本轮改造后）：

| 指标 | 修复前(对冲形同虚设) | 上轮修复后 | 本轮改造后 |
|------|------|------|------|
| 总 PnL | -44,809 | -5,247 | **-3,467** |
| 场均 | -48.0 | -5.6 | **-3.72** |
| 赢场均盈 | +33 | +105 | **+62.8** |
| 亏场均亏 | -130 | -62 | **-19.9** |
| Taker 占比 | — | — | **43%** |

本轮改造要点：对冲梯度化 + PnL 驱动停手 + CancelAll 清场 + hedge_fired_this_tick 修复 + 恢复摽齐 cap + 禁用循环做市。

关键发现：循环做市（对冲步数用完→回退→再对冲）回测显示反而有害（多轮场均 -12.3 vs 单轮 -1.2），已暂停。FSM 层保留回退规则但 engine 不触发，未来条件成熟时可重新启用。

## 待讨论问题

| # | 问题 | 背景 |
|---|------|------|
| 1 | ~~**对冲 Taker 在单 tick 内连环打光 5 步预算**~~ | ✅ 已修复。问题：纯成交驱动下 Taker 即时成交后立刻触发下一步，5 步全在同一秒以同一个 best_ask 成交，没有价格分散。修复：加 `hedge_fired_this_tick` 标记，每个行情 tick 最多打 1 步，BookUpdate 时重置。 |
| 2 | ~~**对冲池预算跨阶段不追踪（200% bug）**~~ | ✅ 已修复。问题：Dynamic 阶段最多花对冲池 100%，切换到 Ev 步数归零又能花 100%，合计最大 200% 超出池子上限。修复：加 `hedge_budget_remaining` 字段，跨阶段不清零，预算不够就停。 |
| 3 | ~~**fsm 与 engine 对冲判定逻辑重复且不一致**~~ | ✅ 已修复。问题：`fsm::hedge_boundary_triggered` 和 `engine::lame_side` 都在判断"哪边穿透亏损线"，但 lame_side 缺少 `hedge_min_qty` 门槛检查。修复：统一到 `PositionSnapshot::breached_side()`，两处共用。 |
| 4 | ~~**DynamicHedging 阶段内的升级条件不严谨**~~ | ✅ 已修复。问题：两边同时穿透亏损线进入 DynamicHedging 时，`both_sides_negative()` 必然为 true 但还要等 count 累加两次才升级，多等一轮没意义。修复：双边同时穿线时直接跳 EvHedging。 |
| 5 | ~~**double_negative_count 不影响执行行为**~~ | ✅ 已确认设计如此。count 仅用于升级判定（给动态对冲第二次机会），不影响对冲执行力度。已在注释中说明意图。 |
| 6 | ~~**strategy crate 命名与实际职责不符**~~ | ✅ 已修复。问题：策略逻辑分散在 fsm（状态流转）、engine（对冲编排）、strategy（订单计算）三处，想看策略得到处翻。修复：把 HedgePhase、HedgeDecision、decide_hedge_step、advantaged_side 搬到 strategy，engine 只做编排和记账。现在看策略决策只需看 strategy + fsm。 |
| 7 | **pool_fraction / follow_fraction 缺少"不超池"校验** | ⏸️ 暂缓。做市侧需追踪挂单占用额度（grid_committed），Engine 要维护订单簿，复杂度高收益低。默认配置触顶概率低且有 Cash Guard 兜底。等策略稳定后再考虑。 |
