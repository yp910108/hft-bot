# 系统架构设计

> 高保真复刻 0xe00740bc 循环做市策略的系统架构。策略见 `docs/策略.md`，标杆分析见 `docs/0xe00740bc 策略分析.md`。
> 本次为全面重构，放下旧包袱：保留经过验证的架构思想（事件驱动单写者、纯函数策略、指令列表下发），但数据结构与接口全部按新策略需求从零设计。

---

## 一、设计原则

1. **单写者事件循环**：所有可变状态只有一个写者（Engine），串行处理事件，无锁无竞态。回测与实盘共用同一套同步核心逻辑，异步只在最外层。
2. **纯函数策略**：策略只做「读世界 → 算决策」，不碰 IO、不分配 ID、不改账本。副作用全部收在 Engine。策略可单测、可确定性回放。
3. **指令列表下发**：策略产出指令意图，Engine 落地成真实指令交给执行后端。策略不持有交易所引用。
4. **逐笔持仓为一等公民**：账本以「每笔买入独立跟踪」为核心数据模型，聚合量是派生视图。这是 LIFO 逐笔止盈的基石，也是与旧架构最大的不同。
5. **买卖对称**：订单、撮合、账本三层都原生支持买入和卖出，不把卖出当特例。
6. **精确数值**：金额/价格/股数全用 `rust_decimal::Decimal`，禁用 f64。
7. **依赖单向**：crate 间依赖无环，领域类型在最底层，编排在最上层。

## 二、Crate 划分与依赖

```
                    ┌─────────────┐
                    │  backtest   │  回测执行 + 数据加载 + 报告
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │   engine    │  单写者事件循环，编排一切
                    └──┬───┬───┬──┘
          ┌────────────┘   │   └────────────┐
   ┌──────▼─────┐   ┌──────▼─────┐   ┌──────▼─────┐
   │  strategy  │   │  exchange  │   │  inventory │
   │ 纯函数决策  │   │  撮合后端   │   │  逐笔账本   │
   └──────┬─────┘   └──────┬─────┘   └──────┬─────┘
          └────────────┬───┴────────────────┘
                　┌─────▼─────┐
                　│  domain   │  领域类型（最底层，无依赖）
                　└───────────┘
```

依赖方向严格向下，无环。各 crate 职责：

| Crate | 职责 | 依赖 |
| --- | --- | --- |
| **domain** | 领域类型：数值、方向、订单、成交、盘口、时钟、精度约束、结算 | 无 |
| **inventory** | 逐笔持仓账本：每笔买入独立记录，支持 LIFO 匹配、卖出减仓、聚合视图 | domain |
| **exchange** | 撮合后端抽象 + 模拟器：买卖双向 Maker/Taker 撮合、挂单簿、撤单、成交回报 | domain |
| **strategy** | 纯函数策略：三阶段决策（建仓/循环做市/收手），逐笔止盈止损规则 | domain, inventory |
| **engine** | 单写者事件循环：持有全部状态，组装上下文 → 决策 → 落地，管理订单生命周期与持仓关联 | domain, inventory, exchange, strategy |
| **backtest** | 回测执行器 + CSV 加载 + PnL 结算 + HTML 报告（含卖出转对面买入的展示层） | domain, engine, exchange, inventory |

**去掉的旧 crate**：
- `risk`（四池风控）：新策略单池，资金约束简化为 Engine 内的现金哨兵，不需要独立 crate。
- `ledger`（聚合账本）：被 `inventory`（逐笔账本）取代。
- `fsm`（状态转移校验）：新策略三阶段是线性推进，转移规则简单，内联到 domain 的状态定义即可，不需要独立 crate。

**新增/改造**：
- `inventory` 新建，替代 `ledger`，核心是逐笔跟踪。

## 三、领域类型（domain）

### 3.1 基础数值与方向（沿用）

```rust
type Price = Decimal;   // 价格 (0,1)
type Qty   = Decimal;   // 股数
type Money = Decimal;   // 金额

enum Side { Up, Down }              // 市场两侧
enum OrderRole { Maker, Taker }     // 撮合角色
enum OrderDirection { Buy, Sell }   // 买卖方向
enum TimeInForce { Gtc, Ioc }       // 有效期
```

### 3.2 订单与成交（扩展卖出语义）

`Order` 沿用现有结构（order_id / side / direction / price / qty / role / time_in_force / generation）。**关键变化在语义**：`direction: Sell` 是一等公民，卖单挂在高价等 bid 穿越成交。

`Fill` 成交回报的现金流按买卖区分：

```rust
struct Fill {
    order_id: OrderId,
    side: Side,
    direction: OrderDirection,   // Buy / Sell
    role: OrderRole,
    price: Price,
    filled_qty: Qty,             // 净成交股数（买入扣费后入仓 / 卖出减仓）
    cash: Money,                 // 买入=付出现金；卖出=回收现金
    generation: Generation,
}
```

- **买入 Fill**：`cash` = 付出的现金，`filled_qty` = 扣 Taker 费后净入仓股数。
- **卖出 Fill**：`cash` = 回收的现金（= 成交价 × 股数 − Taker 费），`filled_qty` = 减仓股数。

### 3.3 精度约束（沿用 + 新增卖单量化）

`OrderConstraints`：min_order_size / min_notional / price_scale / size_scale，默认 5 股 / $1 / 2 位 / 2 位。买单价格向下量化（不抬高买价），**卖单价格向上量化（不降低卖价）**——新增卖单量化方向。

### 3.4 手续费模型（沿用）

Maker 0%，Taker 4%（已核实），扣股数方式。卖出的 Taker 费从回收现金里扣。

### 3.5 结算

15min 到期，赢家侧净持仓每股 $1，输家侧 $0。
```
PnL = 已实现盈亏 + 结算盈亏
    = inventory.realized_pnl + (赢家侧净持仓 × $1 − 净持仓成本)
```

### 3.6 状态（新三阶段）

```rust
enum Phase {
    Building,    // 开场建仓 (progress 0~8%)
    Cycling,     // 循环做市 (progress 8~83%)
    Harvesting,  // 收手变现 (progress 83~96.7%)
    Settled,     // 停手等结算 (progress >= 96.7%)，终态
}
```

线性推进：Building → Cycling → Harvesting → Settled，由 progress 驱动，无回退。转移规则简单，不需独立 fsm crate。

## 四、逐笔持仓账本（inventory）——核心创新

这是新架构与旧架构的根本区别。旧 `ledger` 只有 `SideBook { qty, cost }` 聚合量，无法支撑「每笔买入涨了就卖那一笔」的 LIFO 止盈。

### 4.1 数据模型

```rust
/// 一笔尚未平掉的买入持仓。
struct Lot {
    lot_id: LotId,        // 唯一标识，关联其止盈卖单
    buy_price: Price,     // 这笔的买入价（止盈止损的基准）
    qty: Qty,             // 剩余未平股数
    opened_at: Millis,    // 买入时刻（持有时长、诊断用）
}

/// 单侧的逐笔持仓：一个 Lot 序列。
struct SideInventory {
    lots: Vec<Lot>,       // 按买入时间顺序排列，LIFO 从尾部匹配
}

/// 双边逐笔账本。
struct Inventory {
    up: SideInventory,
    down: SideInventory,
    realized_pnl: Money,  // 累计循环卖出的已实现盈亏
    cash_out: Money,      // 累计买入付出现金
    cash_in: Money,       // 累计卖出回收现金
}
```

### 4.2 核心操作

| 操作 | 语义 |
| --- | --- |
| `open_lot(side, price, qty, at)` | 买入成交 → 新增 Lot，累加 cash_out，返回 lot_id |
| `close_lot(lot_id, sell_price, cash_in)` | 指定 Lot 平仓 → 移除，累加已实现盈亏与 cash_in |
| `partial_close(lot_id, qty, sell_price, cash_in)` | 部分平仓 → 减少 Lot.qty |
| `net_qty(side)` | 某侧净持仓 = Σ lots.qty |
| `net_cost(side)` | 某侧净成本 = Σ (lot.buy_price × lot.qty) |
| `net_avg(side)` | 某侧净均价 = net_cost / net_qty |
| `sum_avg()` | up.net_avg + down.net_avg（结算稳赚判据 < 1） |
| `open_lots(side)` | 遍历某侧未平 Lot（策略挂止盈单用） |
| `net_invested()` | cash_out − cash_in（现金哨兵与结算用） |
| `snapshot()` | 聚合快照（qty/cost 两侧），供结算与报告 |

**LIFO 止盈的实现**：策略遍历 `open_lots(side)`，对每个 Lot 判断「当前 bid 是否 ≥ buy_price + tp」，是则挂一个卖 Lot.qty 的 Maker 卖单，卖单成交后 Engine 调 `close_lot(lot_id, ...)` 平这笔。

### 4.3 已实现盈亏与净投入

账本除持仓外累计已实现盈亏（循环卖出赚的差价）与现金进出，因为最终 PnL = 已实现盈亏 + 结算盈亏，净投入 = cash_out − cash_in（供现金哨兵与结算）。

## 五、撮合后端（exchange）——补齐卖单撮合

旧 `Simulator` **只撮合买单**（`is_fillable` 里 `if direction != Buy return false`），新策略靠卖单止盈，必须补齐。

### 5.1 撮合规则（第一版：穿越即成交）

| 订单类型 | 成交条件 | 成交价 |
| --- | --- | --- |
| Maker 买单 | 卖一价 `best_ask` < 挂单价（严格穿越） | 挂单价 |
| **Maker 卖单** | 买一价 `best_bid` > 挂单价（严格穿越） | 挂单价 |
| Taker 买单（IOC） | 卖一价存在且 ≤ 限价上限 | 卖一价 |
| **Taker 卖单（IOC，止损）** | 买一价存在且 ≥ 限价下限 | 买一价 |

严格穿越是保守口径，减少逆向选择的高估。第二版做队列建模精化。

### 5.2 后端抽象

```rust
trait ExchangeBackend {
    fn submit_order(&mut self, order: Order);
    fn cancel_order(&mut self, order_id: OrderId);
    fn cancel_side(&mut self, side: Side);
    fn cancel_all(&mut self);
}

enum ExchangeEvent {
    BookUpdate(MarketSnapshot),
    Filled(Fill),
    Rejected { order_id: OrderId, reason: RejectReason },
    Canceled(OrderId),
    CancelFailed(OrderId),
}
```

`Simulator` 实现此 trait，`submit_order` 按 role 分流：Taker 立即撮合（买用 best_ask、卖用 best_bid），Maker 进挂单簿等行情驱动。同一套接口，实盘后端将来实现同 trait。

## 六、策略（strategy）——三阶段纯函数

### 6.1 接口契约

沿用「Engine 组装只读上下文 → 策略产出决策」的模式，上下文和决策按新策略重新设计。

```rust
/// Engine 每 tick 组装的只读世界切片。
struct DecisionContext<'a> {
    trigger: Trigger,                 // 本 tick 由什么触发
    now: Millis,
    progress: Decimal,                // 场内进度 0~1（分段调参核心）
    market: MarketSnapshot,           // 当前盘口
    inventory: &'a Inventory,         // 逐笔持仓（只读）
    active_orders: &'a [ActiveOrder], // 当前挂单镜像
    free_cash: Money,                 // 可用现金（现金哨兵用）
    constraints: OrderConstraints,
    config: &'a StrategyConfig,       // 全部参数
}

enum Trigger {
    BookUpdate,
    Filled { side: Side, direction: OrderDirection, lot_id: Option<LotId> },
    OrderUpdate,
}

/// 策略产出的决策。
struct Decision {
    commands: Vec<CommandIntent>,   // 要下发的指令
    transition: Option<Phase>,      // 可选的阶段跳转
}

enum CommandIntent {
    SubmitBuy  { side: Side, price: Price, qty: Qty, role: OrderRole, tif: TimeInForce },
    SubmitSell { lot_id: LotId, price: Price, qty: Qty, role: OrderRole, tif: TimeInForce },
    Cancel(OrderId),
    CancelSide(Side),
    CancelAll,
}
```

**关键设计**：`SubmitSell` 携带 `lot_id`，把卖单与它要平的那笔买入绑定。Engine 落地时记住这个关联，卖单成交后知道平哪个 Lot。这是逐笔止盈能落地的关键。

### 6.2 三阶段策略

每阶段一个纯函数 `decide(ctx) -> Decision`，由 progress 路由：

```rust
trait PhaseStrategy {
    fn decide(&self, ctx: &DecisionContext) -> Decision;
}

struct BuildingStrategy;    // 开场双边铺 Maker 买单
struct CyclingStrategy;     // 循环：补买 + 逐笔挂止盈卖单 + 止损
struct HarvestingStrategy;  // 收手：停买、继续止盈、停手
```

**路由**（Engine 每 tick 按 progress 选阶段）：
```
progress < 8%   → Building
8% ~ 83%        → Cycling
83% ~ 96.7%     → Harvesting
>= 96.7%        → Settled（Skip，只等结算）
```

**CyclingStrategy 的核心逻辑**（每 tick）：
1. **止盈扫描**：遍历 `inventory.open_lots(side)`，对每个尚未挂止盈单的 Lot，若 `bid >= buy_price + tp(progress)`，产出 `SubmitSell { lot_id, price: buy_price+tp, ... }`
2. **止损扫描**：遍历 open_lots，若 `buy_price - bid >= sl(progress)`，撤该 Lot 的止盈单、产出 Taker 卖单止损
3. **补买**：若某侧无足够挂单，在 bid 挂 Maker 买单（优先便宜侧）

止盈幅度 `tp(progress)` 和止损幅度 `sl(progress)` 是 progress 的分段函数，参数在 `StrategyConfig`。

### 6.3 参数配置

```rust
struct StrategyConfig {
    entry_second: u64,           // 入场秒数（6）
    lot_qty: Qty,                // 单笔股数（10）
    building_end: Decimal,       // 建仓期结束 progress（0.08）
    harvest_start: Decimal,      // 收手期开始 progress（0.83）
    settle_start: Decimal,       // 完全停手 progress（0.967）
    tp_by_quartile: [Price; 4],  // 止盈幅度分段 [0.05,0.06,0.08,0.10]
    sl_by_quartile: [Price; 4],  // 止损幅度分段 [0.04,0.04,0.05,0.07]
    // 第二版可选项（默认关闭）：
    inventory_cap: Option<Qty>,       // 单侧净持仓上限
    imbalance_cap: Option<Qty>,       // 双侧不平衡上限
    symmetry_threshold: Option<Qty>,  // 收手对称补仓阈值
}
```

## 七、引擎（engine）——单写者编排

### 7.1 持有的状态

```rust
struct Engine {
    config: EngineConfig,

    // 三阶段策略（无状态纯函数，构造一次复用）
    building: BuildingStrategy,
    cycling: CyclingStrategy,
    harvesting: HarvestingStrategy,

    // 可变状态
    phase: Phase,                        // 当前阶段
    inventory: Inventory,                // 逐笔账本
    book: OrderBook,                     // 活跃挂单镜像
    id_gen: OrderIdGenerator,
    generation: Generation,
    market: MarketSnapshot,

    // 订单 ↔ Lot 关联（卖单成交后知道平哪笔；防重复挂止盈单）
    sell_order_lot: HashMap<OrderId, LotId>,
    lot_sell_order: HashMap<LotId, OrderId>,

    now: Millis,
    time_to_expiry: Millis,
}
```

### 7.2 事件处理主流程

```rust
fn handle_event(&mut self, event, now, tte) -> Vec<Command> {
    self.now = now;
    self.time_to_expiry = tte;
    let trigger = self.apply_event_facts(&event);  // ① 更新账本/挂单/行情
    let decision = self.decide(trigger);            // ② 组装上下文→路由→决策
    self.apply_decision(decision)                   // ③ 落地：分ID、过现金哨兵、更新关联
}
```

**① apply_event_facts**：
- `BookUpdate` → 更新 market
- `Filled(买入)` → `inventory.open_lot(...)`，book 减 qty
- `Filled(卖出)` → 查 `sell_order_lot[order_id]` → `inventory.close_lot(lot_id, price, cash)`，清理关联
- `Canceled/Rejected` → book 移除，清理 Lot↔卖单关联

**② decide**：算 progress，按 progress 路由到对应阶段策略，组装 `DecisionContext`（含 `inventory` 只读引用）。

**③ apply_decision**：
- `SubmitBuy` → 分配 ID、过现金哨兵、入 book、产出 Command
- `SubmitSell { lot_id }` → 分配 ID、入 book、**记录 `sell_order_lot[id]=lot_id`、`lot_sell_order[lot_id]=id`**、产出 Command
- `Cancel*` → 标记 book、产出 Command
- `transition` → 更新 phase、推进 generation

### 7.3 现金哨兵

新策略单池，不需要旧的四池风控 crate。现金哨兵内联到 Engine：下买单前校验 `free_cash >= 单笔名义`，不足则拒发。`free_cash = 总资金 − 净投入 − 活跃买单名义`。卖单回收现金，不占用哨兵额度。

### 7.4 订单生命周期

沿用旧 `OrderBook` 的成熟设计：`Active ↔ CancelPending`，部分成交减 qty、全成交移除，撤单与成交竞态以回调为准。这块经过验证，直接复用其思想。

## 八、回测（backtest）

### 8.1 执行器

沿用现有 `run_match` 的驱动骨架（逐秒快照 → 喂 Simulator 撮合 → 事件喂 Engine → 指令下发 Simulator），因为它无策略假设。改动：
- 记录逐笔成交时，区分买入/卖出，记录 lot 关联（供报告展示）
- PnL = `inventory.realized_pnl + 结算盈亏`

### 8.2 数据加载（沿用）

`load_market` / `load_dir` 解析 `logs/` 逐秒 CSV，胜负判定 `pm_price > ptb`。直接复用。

### 8.3 HTML 报告（新增展示层转换）

**关键**：报告生成时，把真实的 SELL 操作转换成「对面方向的买入」展示，让报告更直观（与 0xe00740bc 原始报告口径一致）。

```
真实成交：SELL UP @ 0.54
报告展示：BUY DN @ 0.46  (标注 converted)
```

这只是**展示层转换**，不影响底层账本、现金流、PnL 的真实语义。转换规则：`SELL side@price` → `BUY side.opposite()@(1-price)`。

报告内容：逐场 PnL、胜率、场均、赢/亏场均、盈亏归因（按策略文档第七节的五类）、逐笔操作序列（转换后）、sum_avg 演变、波动分组表现。

## 十、与旧架构的关键差异

| 维度 | 旧架构 | 新架构 |
| --- | --- | --- |
| 账本 | `ledger`：聚合 `{qty, cost}` | `inventory`：逐笔 `Vec<Lot>`，支撑 LIFO 止盈 |
| 卖出 | Simulator 不撮合卖单 | 买卖双向撮合，卖单是一等公民 |
| 状态机 | 6 状态 + 独立 fsm crate | 4 阶段线性推进，内联 |
| 风控 | `risk`：四池 + 独立 crate | 单池 + Engine 内现金哨兵 |
