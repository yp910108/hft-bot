## 一、设计哲学

四条地基贯穿全局，所有模块设计都服从它们：

### 1. 单写者无锁（Single Writer）

所有并发输入（行情、成交、撤单确认、定时器到期）先进入一个事件队列，由**唯一**的 Engine 线程串行取出、逐个处理。同一时刻只有它在改核心状态（账本、FSM 状态、活跃挂单簿、计数器）。

- **无数据竞争**：物理上只有一个写者，无需锁，没有死锁、无锁竞争开销、延迟可预测。
- **状态永远一致**：每个事件处理是一个原子步骤，不会被打断到一半。
- **逻辑易推理**：处理一个事件时可放心假设没有别的东西在改状态。
- **可回放可复现**：事件是串行序列，录下即可 100% 重放——WAL 复原与回测可复现的根基。

交易系统的瓶颈是网络与撮合延迟（毫秒级），不是 CPU。单线程每秒处理几十万事件，远超行情速率，多线程并行对本项目无收益、徒增风险。

### 2. 纯函数核心（Functional Core, Imperative Shell）

核心决策逻辑（策略、状态路由、风控判定、盈亏计算）全部是**纯函数**：输入「当前状态 + 事件快照」，输出「新状态 + 待发指令」，不碰时间、网络、随机，不改任何外部状态。副作用（IO、改账本、发单、定时器）全部收拢在 Engine 这层「命令式外壳」。

- 每个决策函数都能脱离运行环境单测：给输入、断言输出。
- 决策与副作用分离，bug 定位快、回归测试稳。

### 3. 依赖注入（一切外部能力 trait 化）

时间、交易所、随机数等外部能力全部抽象成 trait。策略与 Engine 只依赖 trait，回测注入虚拟实现、实盘注入真实实现，**同一套核心代码两边运行**。这是回测结论可外推到实盘的前提。

### 4. 事件溯源（Event Sourcing）

所有输入事件与输出指令落盘成不可变序列。任何一场的行为都能完整重放、逐事件单步调试。比仅落 Fill 更彻底，是排障与审计的终极手段。

## 二、整体数据流

```
   ┌── 行情更新 ──┐
   ├── 成交回报 ──┤
   ├── 撤单确认 ──┼──►┌─────────────┐
   └── 定时器到期 ─┘   │  事件队列    │  (mpsc channel，多生产者单消费者)
                      └──────┬──────┘
                             ▼  单写者串行取事件
            ┌────────────────────────────────────────┐
            │              Engine（命令式外壳）          │
            │                                          │
            │  1. 更新账本 / 活跃挂单簿（应用事件事实）    │
            │  2. 组装 DecisionContext（只读世界快照）    │
            │  3. ──► strategy::router 按优先级链路由 ──┐ │
            │                                         │ │
            │     ┌───────── strategy（纯函数核心）────▼─┐
            │     │  router → 选中某阶段小策略.decide()   │
            │     │  返回 Decision{ 指令, 跳转请求 }      │
            │     └────────────────────┬────────────────┘
            │  4. 风控审计 Decision 里的指令              │
            │  5. 下发指令给 ExchangeBackend（trait）     │
            │  6. 事件 + 指令落盘（事件溯源 / WAL）        │
            └────────────────────────────────────────┘
                             │
                             ▼
                   ExchangeBackend（trait）
                   ├── Simulator（回测/模拟）
                   └── Polymarket（实盘，后置）
```

**关键解耦**：Engine 与 strategy 之间只靠一对类型通信——`DecisionContext`（Engine 喂给 strategy 的只读世界切片）和 `Decision`（strategy 还给 Engine 的决策结果）。strategy 不知道指令怎么发、账本怎么记，Engine 不知道决策怎么算。

## 三、Crate 划分（单向依赖）

```
app → backtest/engine → {strategy → fsm, risk, ledger, exchange} → domain
```

domain 零 IO 依赖，被所有人依赖；依赖箭头严格单向，无环。

| Crate | 职责 | 性质 |
|-------|------|------|
| **domain** | 领域模型：数值类型、Side、Order/Fill/Command、TimeInForce、Generation、PnL（结算+浮亏）、EV、FeeModel、MarketSnapshot、RobotState、OrderConstraints | 纯逻辑 |
| **ledger** | 账本（双边持仓/成本/均价，按侧可查）+ WAL 原子落盘与重启复原 | 纯逻辑 + 落盘 |
| **fsm** | 状态定义与合法转移校验：7 状态，不含决策逻辑（决策在 strategy） | 纯逻辑 |
| **strategy** | **纯决策核心**：全局条件路由 + 各阶段小策略，每阶段一文件 | 纯函数 |
| **risk** | 风控审计：四资金池 + Cash Guard + 最大敞口校验 | 纯逻辑 |
| **exchange** | 执行后端 trait + Simulator（Maker 保守填单 / Taker IOC 吃单到保护价）+ Clock trait | trait + 副作用 |
| **engine** | 单写者事件循环 + 账本/挂单簿维护 + 定时器 + 时钟 + 事件溯源落盘 | 命令式外壳 |
| **backtest** | 验证工具：真实 CSV 行情加载 + 虚拟时钟驱动 + 批量统计 | 工具 |
| **app** | 可执行入口 + 配置加载 + 日志 + 真实时钟/真实后端装配 | 装配 |

## 四、strategy crate：小策略组装

策略模式 + 阶段路由。每个阶段是一个独立的纯函数小策略，共享同一接口；router 按优先级链决定每个 tick 归谁管。

### 文件结构

```
strategy/
├── lib.rs              # 门面：模块声明 + 重导出，无实质逻辑
├── context.rs          # DecisionContext / Decision / 共享只读类型
├── router.rs           # 全局条件路由：优先级链裁决当前归哪个小策略
├── building.rs         # 建仓态
├── pairing.rs          # 配对态（含精细订单管理：偏差≤0.01 不撤、缺口变大追加）
├── dynamic_hedge.rs    # 动态对冲（Maker 织网补亏损侧）
├── ev_hedge.rs         # EV 对冲（IOC Taker 顺势追优势方）
└── circuit_breaker.rs  # 熔断求生
```

### 统一接口

```rust
/// 一个阶段小策略：给定只读世界快照，算出此刻的决策。纯函数，无副作用。
trait PhaseStrategy {
    fn decide(&self, ctx: &DecisionContext) -> Decision;
}
```

- **DecisionContext**：只读世界切片——双边持仓/成本/均价、当前盘口、剩余时间 TTE、四池剩余预算、活跃挂单清单、各计数器、上次对冲时间戳。Engine 每个 tick 组装好喂进来。
- **Decision**：本次决策结果——待发指令列表（下单/撤单）+ 可选的阶段跳转请求。strategy 只表达意图，发不发、怎么发由 Engine 落地。

### router 与优先级链

router 是策略文档「全局条件路由（Global Evaluation Router）」的代码化身。每个 tick 从上到下走优先级链，命中即把控制权交给对应小策略：

```
时间红线(<1min) > 熔断求生(Spread>30%) > EV对冲 > 动态对冲 > 核心做市(建仓/配对)
```

router 放在 strategy（优先级链是策略规则，非编排机制），其输入含当前 FSM 状态。Engine 只执行 router 给出的结论。

### 好处

- 每文件短、单一职责，想看哪个阶段只读哪个文件，文档与代码一一对应。
- 每个小策略纯函数，构造 context、断言 decision 即可单测，无需启动 Engine 或交易所。
- 加新阶段 = 加一个文件 + router 插一行，其余阶段零改动。

## 五、engine crate：命令式外壳

```
engine/
├── lib.rs           # 门面
├── event_loop.rs    # 单写者事件循环主体：取事件 → 更新事实 → 路由 → 应用 → 落盘
├── book.rs          # 活跃挂单簿 + 订单生命周期状态机
├── timer.rs         # 定时器：到期往事件队列注入 TimerFired
├── apply.rs         # 落地 Decision：过风控 → 发指令 → 更新本地镜像
└── journal.rs       # 事件溯源：输入事件 + 输出指令落盘
```

### 活跃挂单簿与订单生命周期

新策略的精细订单管理（配对态对比新旧单、最大敞口汇总活跃挂单金额、深海死单撤销、Spread 监测）要求 Engine 持有完整的活跃挂单镜像，而非仅 pending 单。

每笔挂单维护生命周期状态，以**交易所回调为准**（不以本地意图为准）：

```
Active ──(发出撤单请求)──► CancelPending ──┬──(收到 Canceled)──► 移除
                                          └──(收到 Fill)─────► Filled 入账
```

「发出撤单」只是意图，≠「撤单成功」。撤单与成交撞车时，交易所串行处理、命运唯一：要么先成交（撤单失败 + 收 Fill），要么先撤单（收 Canceled、永不成交）。成交是不可撤销的事实，收到 Fill 一律入账。

### 定时器如何融入单写者

3 秒超时、冷却等需要「到点主动触发」。定时器到期**不直接改状态**，而是往事件队列注入一个 `TimerFired` 事件，Engine 仍单写者串行消费。回测中定时器由虚拟时钟驱动。单写者无锁架构毫发无损。

## 六、时间抽象（Clock trait）

新策略处处依赖时间：TTE（剩 5min/1min）、单步 3 秒超时、1/2 秒冷却、熔断 5 秒迟滞。

```rust
trait Clock {
    fn now(&self) -> Instant;       // 当前时刻
}
```

- **回测**：虚拟时钟，由行情 CSV 时间戳推进——可复现、可加速。
- **实盘**：系统 wall-clock。

所有时间逻辑只依赖 Clock trait，策略与 Engine 不关心时间来源。冷却用「时间戳卡点」被动校验（记 s1，后续 tick 比对 `now − s1`）；超时用定时器主动注入 TimerFired 兜底。

## 七、资金池结构（四池）

总资金 $V$ 一级切分由 `CapitalPools` 管理，二级切分由各小策略配置参数控制。

```
总资金 V (100%)
├── 备用金池 Reserve (25%)        日常禁动；Cash Guard 红线 ≥ 此比例；EV 也绝不触碰
├── 核心做市池 Grid_Maker (15%)   建仓三档(2%/3%/5%) + 配对续挂(2%)，均相对本池
├── 动态对冲池 Dynamic (22.5%)    动态对冲单步 = 本池剩余 × 7.5%，三档 40%/30%/30%
└── EV 对冲池 Ev (37.5%)          EV 单步 = 本池剩余 × 25%，IOC@0.85
```

| 约束 | 说明 |
|------|------|
| 四池比例之和 = 1 | `CapitalPools::new` 构造时校验 |
| cash_guard_ratio ≥ reserve | `RiskAuditor` 构造时校验 |
| 最大敞口 ≤ 动态对冲池的 50%（11.25%V） | 风险敞口与救火弹药的硬链接 |
| 各池预算跨阶段不重置 | 花完即停，不归零 |
| 二级 fraction 相对各自池、非总资金 V | 建仓/续挂相对核心做市池；对冲相对各对冲池 |

## 九、回测与验证体系

### 数据源

- `logs/`：933 个真实逐秒盘口 CSV（BTC 15 分钟市场）。
- 胜负判定：末个有效 `pm_price > ptb` → Up 赢。

### 流程

```bash
cargo run -p backtest --example report_json --release   # 跑回测产出 JSON
python3 analysis_data/render_rust_report.py             # 渲染逐场 HTML
```

回测靠虚拟时钟（行情时间戳驱动）注入，使 TTE / 超时 / 冷却 / 熔断迟滞全部可复现。`BacktestReport` 含 final_state 分布，用于诊断各阶段触达率。
